# 5.0.0 — Management Tools

Management tools let an LLM running inside a ForgeBox task manage
ForgeBox itself. Today only **agents** are managed: list, get, create,
update, delete, plus read-only helpers for providers and models. The
mechanism (in-VM execution + per-task token + destructive-action
confirmation) is generic and can host more managed resources later.

## 5.1.0 — Tool surface

Seven tools, registered as native `sdk.ToolPlugin`s. Each tool is
allow-listed on a per-agent basis via the existing `agents.tools` JSON
array (1.1.1, 1.2.3) — an agent that does not list a tool name cannot
call that tool.

| Tool                       | Class | `IsDestructive(input)` | Confirmation? |
|----------------------------|-------|------------------------|---------------|
| `list_agents`              | read  | `false`                | no            |
| `get_agent`                | read  | `false`                | no            |
| `create_agent`             | write | `false`                | no            |
| `update_agent`             | write | `true`                 | yes           |
| `delete_agent`             | write | `true`                 | yes           |
| `list_providers`           | read  | `false`                | no            |
| `list_models_for_provider` | read  | `false`                | no            |

`update_agent` is destructive in the spec sense: it overwrites
user-curated state without the user's consent in the moment.

### 5.1.1 — Input schemas

(Listed in full in `docs/superpowers/specs/2026-05-01-management-tools-design.md`
§3.1. The required fields and enums match the request bodies the
gateway already accepts on `POST/PUT /api/v1/agents`.)

### 5.1.2 — Output shape

Read tools return JSON arrays/objects mirroring the existing REST
responses. `delete_agent` returns `{ "deleted": true, "id": "<id>" }`.
Errors surface to the LLM as `ToolExecResult{ IsError: true }` carrying
the gateway's `error` string.

## 5.2.0 — Execution model

Tool bodies execute inside the Firecracker microVM (architecture rule
#1) and reach the gateway via authenticated HTTPS:

```
LLM → engine.Run → orchestrator.Execute(vmID, name, input)
                        │
                        ▼
                    in-VM tool body
                        │  HTTPS w/ FORGEBOX_API_TOKEN
                        ▼
                    gateway /api/v1/agents (existing handlers)
                        │
                        ▼
                    1.2.2 visibility + 1.2.3 validation (unchanged)
```

### 5.2.1 — Network egress

`vm.AllocRequest` carries a `ControlPlaneAccess bool`. The engine sets
it to `true` when the agent's `tools` array contains any of the seven
tool names above. A VM with `ControlPlaneAccess: true` can reach
`FORGEBOX_API_URL` only — it does **not** gain general internet egress
(rule #3 stays intact).

## 5.3.0 — Identity and auth

The engine issues a per-task opaque token at VM allocation:

- Format: `fbtask_<32-byte-base64url>`.
- Lifetime: bounded by the task timeout; revoked when the task ends.
- Storage: in-memory only on the gateway, lost on restart.

The token and `FORGEBOX_API_URL` are injected into the guest as env
vars. Tool bodies set `Authorization: Bearer <token>` on every request.
The gateway's `(*Server).userID` recognizes the `fbtask_` prefix and
resolves the token via `tasktoken.Store` (shared with the engine) to
the originating user. A bearer token whose value starts with `fbtask_`
but does not resolve returns an empty user id; agent handlers respond
with HTTP 401. Spec 1.2.2/1.2.3 then enforce permissions — no new
authorization layer is added.

## 5.4.0 — Destructive-action confirmation

When a tool's `IsDestructive(input)` returns `true`, the engine pauses
between permission check and `orchestrator.Execute`:

1. Generate a random `approval_id`.
2. Emit `Event{Type: "tool_pending_approval", ToolCall, ApprovalID}`.
3. Wait up to 60 s for the dashboard to send back
   `{type: "tool_approval", approval_id, decision: "approve"|"deny"}`
   via the existing WebSocket.
4. Timeout, `ctx.Done()`, or `decision: "deny"` → tool result is
   `IsError: true, Content: "User declined to approve this action."`
   and the loop continues with no execution.
5. `decision: "approve"` → execute as normal.
6. Emit `Event{Type: "tool_approval_resolved", ApprovalID, Approved}`
   either way.

### 5.4.1 — Gateway → dashboard bridge

The gateway runs the engine asynchronously per `POST /api/v1/tasks` and
sets `task.EventSink` to a buffered channel it drains in a forwarder
goroutine. Two engine event types are republished onto the in-process
events bus (`internal/events`) with the task owner's `UserID` so the
WS hub fans them out only to the originating user's connected clients:

- `task.tool_pending_approval` — payload `{task_id, approval_id, tool_call}`.
- `task.tool_approval_resolved` — payload `{task_id, approval_id, approved}`.

When `engine.Run` returns the gateway closes the sink and waits for the
forwarder to drain so no goroutine outlives the task. Other engine
event types (text deltas, raw `tool_call`, `tool_result`) are not
forwarded here — text streaming reaches the dashboard through the
provider's existing `task.token` path.

The dashboard subscribes to both bus types via the WebSocket and
renders `ToolApprovalCard` for pending approvals; clicking
Approve / Deny dispatches a `{type: "tool_approval", payload:
{approval_id, decision}}` frame, which `(*Server).handleWS` routes to
`engine.Approvals.Resolve`.

## 5.5.0 — Allow-listing

These tools obey the existing per-agent allow-list mechanism: the tool
name must appear in the agent's `tools` JSON array (1.2.3) for the
agent to call it. There is no implicit "managers can do anything"
shortcut.

ForgeBox does not yet have a "default chat agent" concept; how the
dashboard ensures the chat agent has the right tool names allow-listed
is left to the operator (e.g. include them when creating the agent).

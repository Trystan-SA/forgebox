# 3.0.0 — Providers

LLM providers are the integration boundary between ForgeBox and external
inference backends (Anthropic API, Claude Max subscription, OpenAI, Ollama,
etc.). This chapter covers how providers are configured, persisted, and
served to the rest of the system.

## 3.1.0 — Sources of provider configuration

A provider may originate from one of two sources:

### 3.1.1 — Config-file (built-in)

Defined under `providers:` in the server config. Loaded at startup by
`Registry.LoadBuiltins`. Cannot be added, edited, or deleted at runtime via
the API. Useful for static deployments where credentials live in
environment-managed config.

### 3.1.2 — DB-backed (runtime)

Created via `POST /api/v1/providers` from the dashboard or API. Persisted to
the `providers` table, with the credential portion AEAD-encrypted at rest.
Loaded into the registry at startup by `Registry.LoadFromStore` and
re-loaded immediately after creation, so a freshly added provider is usable
without restarting the server. Removable via `DELETE /api/v1/providers/{id}`.

## 3.2.0 — Storage and encryption

### 3.2.1 — Schema

ForgeBox storage runs on PostgreSQL only. The `providers` table holds one
row per DB-backed provider:

| Column             | Type         | Notes                                                  |
|--------------------|--------------|--------------------------------------------------------|
| `id`               | TEXT PK      | UUID assigned on create                                |
| `type`             | TEXT         | Factory key (e.g. `anthropic-subscription`)            |
| `name`             | TEXT UNIQUE  | User-supplied display label; doubles as registry key   |
| `config_encrypted` | TEXT         | base64url(AEAD-sealed JSON of the provider config map) |
| `created_at`       | TIMESTAMPTZ  |                                                        |
| `updated_at`       | TIMESTAMPTZ  |                                                        |

`name` is globally unique across all DB-backed providers; conflicts with
config-file built-ins are also rejected at create time.

### 3.2.2 — Encryption at rest

Credentials never reach the database in plaintext. The `config` map sent in
`POST /api/v1/providers` is JSON-marshaled and sealed with AES-256-GCM by
`internal/crypto.SecretBox`. The envelope format is

```
base64url( version(1B) || nonce(12B) || ciphertext_with_tag )
```

Version 1 is current; older envelopes remain decryptable across key-format
upgrades. The 32-byte master key is loaded from the
`FORGEBOX_DB_ENCRYPTION_KEY` environment variable, accepting either 64 hex
chars or standard/raw base64. The server refuses to start if the key is
missing or malformed.

### 3.2.3 — Key rotation

Out of scope for 3.0.0 — rotation requires re-sealing every existing row.
Documented as future work.

## 3.3.0 — Registry semantics

### 3.3.1 — Registry keys

The registry indexes providers by their **registered name**:

- For config-file built-ins, the key is `p.Name()` (the provider's
  type-name, e.g. `"anthropic"`).
- For DB-backed providers, the key is the user-supplied display name from
  the `name` column.

Names are unique across the entire registry. A `POST /providers` whose
`name` collides with any existing key returns `409 Conflict`.

### 3.3.2 — `PluginMeta` for the dashboard

`GET /api/v1/providers` returns one `PluginMeta` per registered provider,
including:

- `name` — the registry key
- `id` — the DB row id (empty for built-ins)
- `provider_type` — the factory key (so the UI can show the type even when
  the display name differs)
- `builtin` — `true` for config-file entries, `false` for DB-backed

## 3.4.0 — Lifecycle of a runtime-added provider

1. The dashboard POSTs `{ type, name, config }` to `/api/v1/providers`.
2. The gateway constructs a throwaway plugin from `type` and calls `Init`
   with `config`. Validation errors from the provider (bad key prefix,
   missing fields, etc.) surface as `400 Bad Request`.
3. On successful Init, `config` is JSON-encoded, encrypted, and inserted as
   a `providers` row.
4. The same `config` is then used to instantiate a second plugin which is
   registered under `name`. Failure here rolls back the row so the next
   attempt with the same name is not blocked.
5. The handler responds `201 Created` with the new `PluginMeta`.

## 3.5.0 — Deletion

`DELETE /api/v1/providers/{id}` removes the row and unregisters the
matching plugin from the registry, calling `Shutdown` first. Built-ins
(`id == ""`) are not deletable through this endpoint.

## 3.6.0 — Operational notes

- The `FORGEBOX_DB_ENCRYPTION_KEY` env var is required even when no DB
  providers exist — the gateway refuses to register the POST/DELETE routes
  without it. Generate one with:
  ```
  openssl rand -hex 32
  ```
- Backing up the SQLite DB without backing up the encryption key produces
  unusable data. Operators are responsible for treating the key as a
  separate secret.
- The dashboard never displays a stored credential after creation. The
  only way to recover a token is to re-create the provider.

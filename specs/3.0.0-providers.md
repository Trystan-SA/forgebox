# 3.0.0 — Providers

LLM providers are the integration boundary between ForgeBox and external
inference backends (Anthropic API, Claude Max subscription, OpenAI, Ollama,
etc.). This chapter covers how providers are configured, persisted, and
served to the rest of the system.

## 3.1.0 — Sources of provider configuration

A provider may originate from one of two sources:

### 3.1.1 — Config-file (built-in)

Defined under `providers:` in the server config. Loaded at startup by
`Registry.LoadBuiltins`. Cannot be added, edited, or deleted at runtime via
the API. Useful for static deployments where credentials live in
environment-managed config.

### 3.1.2 — DB-backed (runtime)

Created via `POST /api/v1/providers` from the dashboard or API. The request
carries only `type` and `config`; the display name is **derived from the
type** by the gateway (e.g. `anthropic-api` → `Anthropic (API)`,
`anthropic-subscription` → `Anthropic (Subscription)`,  `openai` → `OpenAI`).
Operators do not customize the display name, and one DB-backed provider
per type is allowed — a second `POST` with an already-configured type
returns `409 Conflict`.

Persisted to the `providers` table, with the credential portion AEAD-encrypted
at rest. Loaded into the registry at startup by `Registry.LoadFromStore` and
re-loaded immediately after creation, so a freshly added provider is usable
without restarting the server. Removable via `DELETE /api/v1/providers/{id}`.

## 3.2.0 — Storage and encryption

### 3.2.1 — Schema

ForgeBox storage runs on PostgreSQL only. The `providers` table holds one
row per DB-backed provider:

| Column             | Type         | Notes                                                  |
|--------------------|--------------|--------------------------------------------------------|
| `id`               | TEXT PK      | UUID assigned on create                                |
| `type`             | TEXT UNIQUE  | Factory key (e.g. `anthropic-subscription`)            |
| `name`             | TEXT         | Derived display label (server-set from type)           |
| `config_encrypted` | TEXT         | base64url(AEAD-sealed JSON of the provider config map) |
| `created_at`       | TIMESTAMPTZ  |                                                        |
| `updated_at`       | TIMESTAMPTZ  |                                                        |

`type` is unique across DB-backed providers — only one row per provider
type. `name` is computed by the gateway from the type at create time and
is not user-editable. Conflicts with config-file built-ins (a built-in
already keyed under the same derived label) are also rejected at create
time.

### 3.2.2 — Encryption at rest

Credentials never reach the database in plaintext. The `config` map sent in
`POST /api/v1/providers` is JSON-marshaled and sealed with AES-256-GCM by
`internal/crypto.SecretBox`. The envelope format is

```
base64url( version(1B) || nonce(12B) || ciphertext_with_tag )
```

Version 1 is current; older envelopes remain decryptable across key-format
upgrades. The 32-byte master key is loaded from the
`FORGEBOX_DB_ENCRYPTION_KEY` environment variable, accepting either 64 hex
chars or standard/raw base64. The server refuses to start if the key is
missing or malformed.

### 3.2.3 — Key rotation

Out of scope for 3.0.0 — rotation requires re-sealing every existing row.
Documented as future work.

## 3.3.0 — Registry semantics

### 3.3.1 — Registry keys

The registry indexes providers by their **registered name**:

- For config-file built-ins, the key is `p.Name()` (the provider's
  type-name, e.g. `"anthropic"`).
- For DB-backed providers, the key is the derived display label from the
  type (e.g. `"Anthropic (API)"` for type `anthropic-api`). The mapping
  lives in `internal/plugins/labels.go`.

Names are unique across the entire registry. A `POST /providers` whose
derived label collides with any existing key — built-in or DB-backed —
returns `409 Conflict`. A second `POST` with an already-configured type
also returns `409`.

### 3.3.2 — `PluginMeta` for the dashboard

`GET /api/v1/providers` returns one `PluginMeta` per registered provider,
including:

- `name` — the registry key
- `id` — the DB row id (empty for built-ins)
- `provider_type` — the factory key (so the UI can show the type even when
  the display name differs)
- `builtin` — `true` for config-file entries, `false` for DB-backed
- `models` — the provider's model catalog (id, display name, token limits,
  capability flags), in the provider's preferred display order

### 3.3.3 — Model ordering

Each provider returns its `Models()` list in **most-powerful-first** order.
The dashboard relies on this contract for two behaviors:

- The shared model selector renders one combined dropdown, with each
  provider's models grouped under an `<optgroup label={provider.name}>`.
  Within a group, models appear in `Models()` order, so the strongest
  model sits at the top of each provider's section.
- When a form has no saved model yet, the dashboard auto-selects the
  first model of the first provider (the strongest available choice).

Plugins must therefore order frontier models above smaller/cheaper
variants (e.g. Opus → Sonnet → Haiku for Anthropic; gpt-4.1 →
gpt-4.1-mini → gpt-4.1-nano for OpenAI).

### 3.3.4 — Chat surface persistence

The dashboard chat (`/dashboard`) persists the user's last chosen
`(provider, model)` to `localStorage` under `forgebox.chat.lastModel` and
restores it on next visit. If the saved pair no longer exists (provider
deleted, model delisted), the standard 3.3.3 default kicks in: snap to
the first provider's first model.

## 3.4.0 — Lifecycle of a runtime-added provider

1. The dashboard POSTs `{ type, config }` to `/api/v1/providers`. The
   request body intentionally omits a name field — it is derived from
   `type` server-side.
2. The gateway looks up the canonical display label for `type` and
   rejects unknown types with `400 Bad Request`. It also rejects the
   request if a DB row of this type already exists, or if the derived
   label collides with a built-in registry key, with `409 Conflict`.
3. The gateway constructs a throwaway plugin from `type` and calls `Init`
   with `config`. Validation errors from the provider (bad key prefix,
   missing fields, etc.) surface as `400 Bad Request`.
4. On successful Init, `config` is JSON-encoded, encrypted, and inserted
   as a `providers` row with `name` set to the derived label.
5. The same `config` is then used to instantiate a second plugin which
   is registered under the derived label. Failure here rolls back the
   row so the next attempt with the same type is not blocked.
6. The handler responds `201 Created` with the new `PluginMeta`.

## 3.5.0 — Deletion

`DELETE /api/v1/providers/{id}` removes the row and unregisters the
matching plugin from the registry, calling `Shutdown` first. Built-ins
(`id == ""`) are not deletable through this endpoint.

### 3.5.1 — Settings page actions

The Providers tab in `/settings` exposes a small icon-only Delete button
in the top-right of each non-builtin card. Clicking it opens an in-app
`<ConfirmModal>` (not a native browser confirm) styled in the danger
variant; on confirm the dashboard `DELETE`s the provider and refreshes
the list. Built-in providers render the chart but no delete button.

The reconfigure flow is just *delete then add a new one* — no dedicated
"revoke" shortcut, since the user can navigate to `/providers/new`
themselves and the form lets them pick any type.

### 3.5.2 — Per-provider usage chart

data source is currently mocked client-side (deterministic per provider

## 3.6.0 — Operational notes

- The `FORGEBOX_DB_ENCRYPTION_KEY` env var is required even when no DB
  providers exist — the gateway refuses to register the POST/DELETE routes
  without it. Generate one with:
  ```
  openssl rand -hex 32
  ```
- Backing up the SQLite DB without backing up the encryption key produces
  unusable data. Operators are responsible for treating the key as a
  separate secret.
- The dashboard never displays a stored credential after creation. The
  only way to recover a token is to re-create the provider.

## 3.7.0 — Task error surfacing

When `engine.Run` returns an error, the gateway persists a short
user-facing string into `TaskRecord.Error` (not the raw wrapped chain).
The mapping is:

| Sentinel (`pkg/sdk/errors.go`) | Stored message |
|--------------------------------|----------------|
| `ErrRateLimit`     | "Rate limit exceeded. The provider is throttling requests — wait a moment and try again, or switch to a different model." |
| `ErrAuth`          | "Provider authentication failed. The configured credentials are missing, expired, or unauthorized — reconfigure the provider in Settings." |
| `ErrInputTooLarge` | "The prompt is too large for this model's context window. Shorten it or pick a model with a larger context." |
| `ErrTransient`     | "The provider is temporarily unavailable. Try again in a moment." |
| (unmatched)        | `err.Error()` (full chain — preserves debuggability for unexpected failures) |

The original wrapped error is still logged via `slog` at the call site so
operators can correlate task IDs with raw provider responses. The chat
dashboard renders `TaskRecord.Error` in its red error block.

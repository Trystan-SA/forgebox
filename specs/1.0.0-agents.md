# 1.0.0 — Agents

Agents are reusable AI personas configured by users. Each agent bundles a
provider/model selection, a system prompt, an allow-list of tools, and a
sharing scope. Tasks invoke an agent (rather than naming a provider+model
inline) so behavior is consistent across runs.

This chapter covers persistence, the REST API, and the rules the gateway
enforces around agent records.

## 1.1.0 — Storage

### 1.1.1 — Schema

ForgeBox storage runs on PostgreSQL only. The `agents` table holds one row
per agent:

| Column          | Type         | Notes                                                                 |
|-----------------|--------------|-----------------------------------------------------------------------|
| `id`            | TEXT PK      | UUID assigned on create                                               |
| `name`          | TEXT         | User-supplied display label (not globally unique)                     |
| `description`   | TEXT         | Free-form description, may be empty                                   |
| `system_prompt` | TEXT         | Instructions injected as the system message, may be empty             |
| `provider`      | TEXT         | Registry key of a configured provider (matches `PluginMeta.name`)     |
| `model`         | TEXT         | Model id from the selected provider's `Models()` catalog              |
| `tools`         | TEXT         | JSON array of tool names the agent is allowed to call                 |
| `sharing`       | TEXT         | `personal`, `team`, or `org`                                          |
| `team_id`       | TEXT NULL    | Required when `sharing = team`; ignored otherwise                     |
| `created_by`    | TEXT         | User id of the creator                                                |
| `created_at`    | TIMESTAMPTZ  |                                                                       |
| `updated_at`    | TIMESTAMPTZ  |                                                                       |

`tools` is stored as JSON text (matching how `apps.tools` is stored). The
gateway is responsible for keeping it in sync with the actual built-in
tool registry — invalid tool names are rejected at create/update time.

### 1.1.2 — No credential storage

Agents reference providers by name; they do not embed credentials. The
provider's encrypted config is the sole secret store (see 3.2.0). This
keeps the `agents` table free of anything that would require AEAD
encryption or rotation.

## 1.2.0 — REST API

### 1.2.1 — Endpoints

| Method | Path                      | Purpose                  |
|--------|---------------------------|--------------------------|
| GET    | `/api/v1/agents`          | List visible agents      |
| POST   | `/api/v1/agents`          | Create a new agent       |
| GET    | `/api/v1/agents/{id}`     | Get one agent            |
| PUT    | `/api/v1/agents/{id}`     | Patch mutable fields     |
| DELETE | `/api/v1/agents/{id}`     | Delete an agent          |

The `/api/v1/agents/{id}/brain/...` endpoints (existing) attach to an
agent's persistent memory; they remain unchanged.

### 1.2.2 — Visibility filter

`GET /api/v1/agents` returns the union of:

- agents created by the requesting user, plus
- agents shared `team` whose `team_id` is in the user's team list, plus
- agents shared `org`.

`personal` agents are visible only to their creator.

### 1.2.3 — Validation on create / update

The gateway validates these constraints before writing:

- `name` is required and non-empty.
- `sharing` defaults to `personal` if absent; must be one of `personal`,
  `team`, `org`.
- When `sharing = team`, `team_id` is required and must be a team the
  caller belongs to.
- `provider`, when set, must match a registered provider name. An empty
  `provider` is allowed at create time (the dashboard saves drafts), but
  task execution will reject the agent until both `provider` and `model`
  are populated.
- `model`, when set, must appear in the named provider's `Models()`
  catalog.
- `tools` is a JSON array of strings; each entry must match a registered
  tool plugin name.

Any violation returns `400 Bad Request` with the failing field named in
the error message.

### 1.2.4 — Default model selection

The dashboard defaults `model` to the first entry in the chosen
provider's `Models()` list (the most powerful model — see 3.3.3). The
backend does not auto-fill `model`; if the client omits it the row is
written with an empty `model`.

## 1.3.0 — Lifecycle and deletion

Deletion is hard delete: the row is removed and any attached brain rows
cascade via `automation_id` foreign keys (brain rows are scoped to
automations, not agents directly, so this is a no-op for now). Tasks that
reference a deleted agent by id will fail at lookup time with
`404 Not Found`.

Renaming a provider does not cascade into the `agents.provider` column.
Operators that rename providers are responsible for any agent-side
fix-up; the registry will surface a clear "provider not found" error at
task-execution time.

The legacy `role` column (formerly `worker`/`orchestrator`) is dropped.
Existing rows are unaffected — no behavior outside storage ever
depended on the value, so the column is removed without a backfill.

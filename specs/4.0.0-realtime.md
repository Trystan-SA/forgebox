# 4.0.0 — Real-time event delivery

ForgeBox delivers server-originated events (chatbot replies, task progress,
workflow notifications) to connected dashboards over a single WebSocket
connection per browser session. This chapter defines the wire protocol,
the connection lifecycle, and the in-process bus that producers publish to.

## 4.1.0 — Transport

### 4.1.1 — One WebSocket per browser session

The dashboard opens a single WebSocket to `GET /api/v1/ws` for the lifetime
of the page. There are no per-resource streams; all events for the
authenticated user are multiplexed onto this one connection. The Vite dev
server proxies `/api/v1/ws` to the backend with `ws: true`.

### 4.1.2 — Wire envelope

Every message in either direction is JSON with the same shape:

```json
{ "type": "<event_type>", "payload": { ... } }
```

`payload` is omitted when there is none. Frames the client cannot parse are
ignored, not error-closed.

### 4.1.3 — Server message types

| Type           | Payload                                  | Meaning                              |
|----------------|------------------------------------------|--------------------------------------|
| `auth_ok`      | _none_                                   | Auth handshake succeeded             |
| `auth_error`   | `{ "error": "..." }`                     | Auth handshake failed                |
| `ping`         | _none_                                   | Server liveness probe (every 30s)    |
| `task.updated` | `{ "task_id": "...", "status": "..." }`  | Task state change                    |
| `task.token`   | `{ "task_id": "...", "delta": "..." }`   | LLM token stream chunk               |
| `notification` | `{ "title": "...", "body": "..." }`      | Workflow/automation notification     |

Additional types may be added as new features land. The envelope never
changes; the routing key on both sides is `type`.

### 4.1.4 — Client message types

| Type   | Payload                | When                          |
|--------|------------------------|-------------------------------|
| `auth` | `{ "token": "..." }`   | Required first message        |
| `pong` | _none_                 | Reply to a server `ping`      |

## 4.2.0 — Authentication

### 4.2.1 — First-message handshake

The connection upgrades unauthenticated. The server waits up to 10 seconds
for a single `auth` message. On timeout or on any other first message, the
connection is closed with WebSocket close code `4001`. Tokens are never
placed in the URL.

### 4.2.2 — Token validation

Token validation matches the existing REST middleware: any non-empty token
is accepted. The token also serves as the fan-out key — clients carrying
the same token receive the same events, regardless of how many tabs they
have open. Cryptographic token validation is a pre-existing gap in the
auth layer and is tracked separately.

## 4.3.0 — Liveness

### 4.3.1 — Server-initiated ping

The server sends a `ping` every 30 seconds. The client must reply with a
`pong` within 10 seconds. If no `pong` arrives in time, the server closes
the connection.

### 4.3.2 — Client reconnect

On any close, the client reconnects with exponential backoff (`1s, 2s, 4s,
8s, 16s, capped at 30s`). The backoff resets on the next successful
`auth_ok`. The dashboard's "Connected/Disconnected" badge reflects the
socket's current `connected` state directly — there is no separate health
poll.

## 4.4.0 — Server-side architecture

### 4.4.1 — In-process event bus

Producers (engine, sessions, future subscribers) publish to a typed
`events.Bus` rather than calling the WebSocket layer directly. The bus
delivers each event to every registered subscriber over a buffered channel.
Delivery is non-blocking: a full subscriber channel drops the event for
that subscriber rather than stalling other subscribers.

### 4.4.2 — Hub fan-out

The WebSocket Hub is the only subscriber by default. It maintains
`map[userID][]*Client` and writes each event onto the send channel of
every client matching the event's `UserID`. A client whose send buffer
fills up is evicted (slow client must not stall the hub).

### 4.4.3 — Single-instance scope

Both the bus and the hub are in-process. Multi-instance fan-out (e.g. via
Redis pub/sub) is not supported and is out of scope for this chapter.
